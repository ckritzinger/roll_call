    edition: 3

    global:

        - SELECT * FROM class_slots
        - SELECT * FROM class_instances
        - SELECT * FROM clients
        - SELECT * FROM client_recurring_slots
        - SELECT * FROM sessions

  Key differences from Sync Rules:
  - config: edition: 3 header is required
  - streams: instead of bucket_definitions:
  - queries: instead of data:
  - auto_subscribe: true means the client syncs this stream automatically on
  connect — no client-side subscription call needed
  - No parameterization needed since this is single-user (global stream)

  I'm deliberately not syncing invoices or settings — neither is used on the
  dashboard and they'd just be dead weight in the local SQLite DB.

  5. PowerSync marks the transaction complete

  PowerSync retries the upload when connectivity returns.

❯ wtf is a Cross-Origin-Opener-Policy                                           
  ⎿  UserPromptSubmit hook error
  ⎿  Failed with non-blocking status code: node:internal/modules/cjs/loader:1424

⏺ When a browser tab opens another tab (or a popup), by default they can
  reference each other's window object. COOP cuts that off — a page with
  same-origin says "no cross-origin window can get a handle to me, and I can't
  get one to them."

  That's it. It has zero effect on fetch/XMLHttpRequest to external APIs like
  Supabase or PowerSync — those keep working normally.

  The reason PowerSync needs it: the SharedWorker that handles sync uses
  SharedArrayBuffer for coordination between threads. Browsers disable
  SharedArrayBuffer on pages that don't have COOP set, because without it a
  malicious cross-origin page could exploit timing attacks via the shared
  memory. COOP is basically the browser's precondition for unlocking that
  feature.

  So: COOP = "this tab is isolated from other windows" → browser trusts it
  enough to enable SharedArrayBuffer → PowerSync's worker can use it.
